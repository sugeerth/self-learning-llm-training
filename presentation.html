stage ①

SFT continuation: extend the winner

Take the sweep winner (4L × 512d, 11.28M params) and train another 1000 steps on the same data with cosine LR decay.

warmup: 100 steps
weight_decay: 0.1
grad_clip: 1.0
batch_size: 8

why this works

Winner was undertrained — sweep budget was 200 steps to compare configs cheaply. With the architecture decision locked, more steps = monotonic ppl improvement until overfitting (~3000 steps for this model size on 1MB data).

stage ②

Synthetic data flywheel

Generate 10k samples from the SFT model, filter aggressively through 3 LLM judges, mix survivors back into training.

why three judges

Multi-judge ensemble: judge of judge of judge

Judge-1 coarse · haiku

Binary classifier: is this Shakespearean style? Cheap, fast, catches obvious garbage.

cost: ~$0.0002 / sample

Judge-2 fine · sonnet+thinking

Score 1-10 with weighted breakdown: style (3) · format (2) · lex (2) · coh (3). Calibrated.

cost: ~$0.003 / sample

Judge-of-Judge · opus+thinking

Audits Judge-2's score against the actual text. Catches collusion + drift over time.

cost: ~$0.025 / sample

stage ③

GRPO — Group Relative Policy Optimization

DeepSeek-R1's RL algorithm. Like PPO but no value model — uses group-relative reward as advantage. Saves ~50% memory.

loss equation — hover each term

L = −𝔼i[ AiGroup-relative advantage. Ai = (Ri − μR) / σR. Replaces the value-network in PPO. Zero-mean, unit-variance across the G samples. · log πθ(yi|x)Policy log-probability of completion yi given prompt x. Sums over completion tokens only (not prompt). This is the "policy gradient" term. ] + βKL coefficient (DeepSeek default = 0.04). Higher β = stay closer to reference. Too high → no learning. Too low → reward hacking. · KL(πθ ‖ πref)KL divergence from current policy to frozen SFT snapshot. Prevents catastrophic forgetting of language modeling capability.
  1. Sample G=4 completions per prompt from πθ
  2. Score each with reward function R (rule-based or judge-LLM)
  3. Group-normalize → Ai
  4. Compute KL to πref (frozen SFT snapshot)
  5. Backprop loss above

hyperparameters

group_size G:    4
kl_coef β:       0.04   (DeepSeek default)
lr:              1e-5   (10× lower than SFT)
max_new_tokens:  64
temperature:     0.9    (sampling diversity)
n_steps:         200
reward:          0.5 * α + 0.5 * cloze_acc
                 where α = alpha-char ratio

why GRPO over PPO

No critic = no second forward pass per step, no advantage estimator to tune, no value-loss weighting. For an 11M model trained on a laptop, this is the difference between "feasible" and "OOM".

Why small reasoning models matter (right now)

cost per AIME problem ($)

★ R1-Distill-1.5B is ~1000× cheaper per problem than GPT-4o for similar math accuracy.

three reasons it works in 2026

  1. Distillation works at extreme ratios. R1 (671B) → R1-Distill (1.5B) loses only ~25 pts on AIME because reasoning traces capture knowledge.
  2. Verifiable rewards exist. Math has a checkable answer. Code has tests. No judge bias, no Goodhart.
  3. Process supervision > outcome supervision. Per-step PRM rewards beat sparse final rewards by 6-9 pts on math benchmarks.
Implication for our pipeline: swap our cloze reward for verifiable math reward, swap Tiny Shakespeare base for SmolLM2-360M, add MCTS+PRM. Same code structure.
PRM vs ORM

Process Reward Models (PRM): score every step

ORM (Outcome Reward Model)

Only the final answer gets a 0/1 reward. The model has no idea WHICH step was wrong.

Q: "John has 5 apples, gives 2..."
Step 1: "5 - 2 = 3"  ← correct
Step 2: "3 + 1 = 5"  ← WRONG
Step 3: "Answer: 5"

Reward signal: 0 (final wrong)
Gradient flows to: ALL tokens equally.
Step 1 gets punished for Step 2's error.

PRM (Process Reward Model) ★

Each step gets its own reward. Surgical credit assignment.

Q: "John has 5 apples, gives 2..."
Step 1: "5 - 2 = 3"  → PRM: 1.0  ✓
Step 2: "3 + 1 = 5"  → PRM: 0.0  ✗
Step 3: "Answer: 5"  → PRM: 0.0

Per-step rewards: [1.0, 0.0, 0.0]
Step 1 keeps positive gradient.
Step 2 gets isolated punishment.

Verifiable rewards: math + code = no Goodhart

Rule-based rewards eliminate judge bias entirely. The reward is checkable, not opinable.

math (GSM8K-style)

def math_answer(generated, gold):
    # extract last number from answer
    nums = re.findall(r"-?\d+(?:\.\d+)?",
                      generated.split("Answer:")[-1])
    if not nums: return 0.0
    try:
        return 1.0 if abs(float(nums[-1])
                           - float(gold)) < 1e-3 else 0.0
    except ValueError:
        return 0.0
Pros: deterministic, fast, no API call.
Cons: only works for problems with clean numeric answers.

code (HumanEval-style)

def code_passes(code, tests):
    passed = 0
    for inp, expected in tests:
        try:
            local = {}
            exec(code, {"__builtins__": __builtins__},
                 local)
            out = local["solve"](inp)
            if str(out) == str(expected):
                passed += 1
        except Exception:
            pass
    return passed / len(tests)
Pros: ground truth, partial credit.
Cons: needs sandboxing in production (subprocess + timeout + restricted env).

auxiliary: format compliance

Even with verifiable rewards, models drift away from <think>...</think><answer>...</answer> format. Add a tiny aux reward: +0.3 if think tags present, +0.4 if answer tags, +0.3 if ordered. Keeps the structure.

Long-CoT training & test-time compute scaling

o1 / R1 secret: train the model to use 8k-32k thinking tokens. Accuracy scales monotonically with token budget at inference.

accuracy vs reasoning tokens (R1 reproduction)

training implications

  • Context length: increase from 128 → 8192. Use RoPE scaling (you already have it!).
  • Group size G in GRPO: drop from 4 → 2 because each completion is now 16× longer.
  • Memory: attention is O(N²). Add gradient checkpointing + FlashAttention-3.
  • Curriculum: start with 512-token CoT, ramp to 8k over training.
  • KV cache size: matters at inference. GQA (n_kv=2) cuts KV cache 4× — you already use this!
Existing repo advantage: our model.py already has RoPE, GQA, MQA. To do long-CoT we just bump max_seq_len + train data.

Why GRPO failed (cloze=0%)

Reward = 0.7 · cloze_match + 0.3 · alpha_ratio. The cloze_match was 0 in every sampled completion (model can't predict "question" zero-shot). So reward variance ≈ 0 → advantages ≈ 0 → no policy gradient. Need either (a) richer reward (LLM judge), (b) easier curriculum, or (c) seed completions with correct answer.

Why speculative was slower

I used the SFT snapshot as draft and the GRPO model as target — both 11.28M params, same architecture. Draft has no compute advantage. K=4 draft calls + verification > 4 baseline calls. Speculative needs draft to be 4-10× smaller. Use the "tiny" sweep variant (2.95M, 256d×4L) as draft instead.

What worked

SFT continuation reduced ppl 25% in 200 steps (~25 sec on MPS). The training pipeline + data loader + agent infrastructure all worked end-to-end without errors. Real numbers are in results.json, served at /api/results.
